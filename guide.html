Complete user guide

LifeOS is a personal operating system for designing an ideal day, executing it, recording what actually happened, and improving the system with evidence instead of relying on memory or motivation.

Installable PWAWorks offline after first loadData stored on this deviceManual JSON backup
Critical data rule: LifeOS does not yet have accounts or cloud sync. Your phone and computer hold separate data. Export a JSON backup before clearing browser data, uninstalling the app, changing phones, or importing a major dataset.

1. Quick start

Use this sequence the first time. Do not configure every possible feature before testing a real day.

Set up your profile and default weekEnter the information needed for planning estimates and classify weekdays as Workday, WFH or Off-Day.
Review starter tags and rulesKeep the useful ones, rename or reorder them, and delete noise.
Create only tomorrow’s essential tasksStart with 8–15 tasks. Add times, tags, priority, recurrence and alarms only where they matter.
Execute from the Today pageMark tasks done or skipped, use progress controls, and record actual quantities or measurements.
Review at day-endCheck completion, skipped tasks, nutrition totals, time use and the gap between the planned and actual day.
Export a backupOnce your configuration is stable, create a JSON backup and store it outside the browser.
Best first-day test

Create a morning medicine task, a 20-minute walk, one work priority, dinner before a target time, and one learning task. Use the app for a full day before adding more complexity.

2. Navigation and modules

The top navigation contains the core planner and optional personal-development modules.

Ideal Day

The operational center: calendar, tasks, filters, progress, trackers, totals and libraries.

Modules

Show or hide optional areas. Core pages remain enabled so the app and its safety controls stay accessible.

English Lab

Build reusable communication assets through phrases, roleplay, stories, networking and self-presentation.

Changelog and About

Review product evolution, limitations, installation guidance and this full guide.

On mobile, use the hamburger menu to open navigation. Hiding a module does not automatically delete its data; it removes the module from normal navigation.

3. Calendar and day types

The selected date controls which task occurrences and daily records you are viewing.

Selecting a date

Use the left calendar on desktop or the calendar/panel control on mobile. The Today button returns to the current date.

Workday, WFH and Off-Day

Each date can follow the default weekly pattern or receive a manual category. Task and alarm rules can behave differently for each category:

BehaviorMeaningExample
ActiveThe task appears and is expected on that day type.Office commute on a Workday.
Auto-skipThe occurrence is automatically treated as skipped when the day type makes it irrelevant.Commute on WFH.
Hidden / inactiveThe occurrence is not presented as an active task.Office-only meeting preparation on an Off-Day.
Scenario: unexpected work from home

Change today from Workday to WFH. Commute-related tasks can auto-skip, while exercise, medication and work priorities remain active.

4. Creating and editing tasks

A task can be simple or structured. Add only the fields that improve execution or later analysis.

Basic fields

  • Title: write a clear action, not a vague theme.
  • Description: include useful context, standards or a definition of done.
  • Time: use when sequence or punctuality matters.
  • Priority: reserve higher priority for genuinely consequential work.
  • Tags: connect the task to life areas such as health, family, career or finance.

Repeat and day behavior

Configure recurrence for routines and use day-type behavior to prevent irrelevant occurrences. One-time tasks should not be disguised as permanent routines.

Task actions

Edit to change the underlying task, duplicate to reuse its structure, and delete carefully. When prompted, distinguish between deleting one occurrence and deleting the repeating task or series.

Good task: “Walk 20 minutes after dinner.” Weak task: “Health.” The first can be executed and measured; the second cannot.

5. Task status, next task and overdue logic

Done

Use when the expected occurrence was completed. Add progress or measurements first where relevant.

Skipped

Use when it was deliberately not completed. Skipping is data; do not mark a task done merely to clean the list.

Next task

Highlights the next actionable item according to the app’s sequence and status logic.

Overdue

A past scheduled time can visually de-emphasize or mark an unfinished task as overdue.

Use status honestly. A truthful skipped record is more useful than a fake completion because later review depends on the integrity of the data.

6. Filters and task search

Filters work together as multiple levels. They should narrow the same task list rather than override one another.

  • Text: matches task titles and relevant text.
  • Status: narrows by states such as pending, done, skipped, overdue or next task.
  • Tag: limits the list to one or more life areas.

On mobile, tap the filter button in the date row to show or hide the filter panel. Applying a filter should not permanently alter tasks; it changes only what is currently visible.

Scenario: find unfinished health tasks

Select the Health/Fitness tag, choose the pending status, then type “walk” or “medicine”. All conditions should apply simultaneously.

7. Tags and tag progress

Tags group tasks into life areas and feed the progress summary.

Creating and editing

Choose a name, icon and color. Avoid overlapping categories such as “Work”, “Career” and “Office” unless each has a deliberate distinction.

Reordering

Drag tags to change their order. Use the drop indicator to see where a tag will land. Reordering should not reset the panel’s scroll position.

Progress

The progress panel summarizes completion and scheduled time by tag. Treat it as a signal, not a score to maximize blindly.

8. Progress, quantities and measurements

Not every task is binary. LifeOS supports more detailed evidence where “done” alone would hide useful information.

Step progress

Use incremental controls for tasks completed in several units or stages.

Quantity tracking

Record amounts such as glasses, pages, repetitions, minutes or servings.

Measurement tracking

Record structured values such as blood pressure components, glucose, weight or other configured measures.

Occurrence notes and rating

Add context or a quality rating when the result matters beyond simple completion.

Health warning: LifeOS records and summarizes information; it does not diagnose illness or replace professional medical advice.

9. Time tracking and focus timer

Use the timer for work where actual duration matters.

Starting and minimizing

Start the task timer from its action controls. The full-screen focus view can be minimized to a floating timer while you continue using the app.

Background limitation

Mobile browsers may suspend the page when the app is closed or the screen is off. A robust timer should calculate elapsed time from saved timestamps when LifeOS resumes rather than depend on a JavaScript counter running every second.

Practical rule: Closing LifeOS should not be treated as guaranteed continuous background execution. Reopen the app and verify the timer before relying on it for formal billing or compliance records.

10. Alarms and reminders

A task can have one or more alarms, each with its own time, label, enabled state, repeat setting and day-type behavior.

While LifeOS is open

The alarm engine can show an in-app alert and use available sound/notification behavior.

When LifeOS is closed

A browser-based PWA cannot guarantee exact alarm delivery after Android suspends it. Exact closed-app alarms require web push infrastructure or a native Android layer.

Missed alarm handling

When reopening the app, review missed or overdue items rather than assuming every scheduled alert fired in the background.

11. Templates and repeatable routines

Templates let you reuse a proven day or task structure without rebuilding it manually.

  • Create templates from stable routines, not temporary experiments.
  • Review times and recurrence after applying a template.
  • Do not duplicate the same repeating task through both a template and an existing recurrence.
  • Keep separate patterns for materially different days, such as office, WFH and off-day routines.

12. Food library and nutrition totals

The food library supports meal logging and daily totals for calories, macronutrients, sugar, added/free sugar, salt and fiber.

Logging food

Select an item, enter the amount and verify the unit. Packaged and homemade foods vary, so edit or create entries when the starter estimate is materially wrong.

Interpreting totals

Total sugar can be high even when added sugar is zero because fruit, milk and other foods contain naturally occurring sugar. Juice sugar is tracked differently from intact fruit because processing changes how it is consumed.

Red warnings

Red or warning states indicate that a configured or reference threshold has been exceeded. They are prompts for review, not a medical diagnosis.

Scenario: “No added sugar” but total sugar is high

Check fruit portions, juice, milk and yogurt. Zero table sugar does not automatically mean low total carbohydrate or low total sugar.

13. Workout library and exercise logging

Use the workout library to standardize repeatable exercise actions and connect them to daily execution.

  • Create clear workout entries with duration, sets, repetitions or intensity where useful.
  • Use task progress to record actual output, not just intent.
  • Increase exercise load gradually and account for medical limitations.
  • Use weekly patterns rather than forcing the same workout every day.

14. Profile and planning estimates

The profile can calculate screening and planning estimates such as BMI, resting energy, maintenance calories, macro ranges and exercise baselines.

These calculations depend on the quality of entered data and use general formulas. They can be wrong for individuals with unusual body composition, pregnancy, breastfeeding, kidney disease, heart disease or other conditions requiring individualized targets.

Do not use LifeOS to change prescribed medication or replace a clinician’s plan. Use it to organize habits, record observations and prepare better questions for professionals.

15. Life rules

Rules are standing principles that guide repeated decisions without becoming scheduled tasks themselves.

Good rule examples

  • Do not delay high-impact customer escalations beyond the same working day.
  • Export a LifeOS backup before any major import or browser reset.
  • Do not use screen time during family dinner.

Weak rule examples

“Be better” and “work hard” are too vague. A useful rule should alter a specific decision or behavior.

Rules can be reordered. Keep the list short enough that you can actually remember and apply it.

17. Optional modules and placeholders

Professional Database, Learning Log, Journal and Frameworks may appear as optional areas depending on the current build.

Some are foundations or placeholders rather than fully mature modules. Do not treat their presence in navigation as proof that every workflow is complete. Enable only what contributes to daily use.

18. Export, import and backup safety

Backup is not optional because current LifeOS data lives in browser storage.

Export

Export a JSON backup regularly and store copies outside the device—for example in a private cloud drive and a computer folder.

Import using Merge

Merge attempts to combine imported records with existing data. Use it when you want to add or reconcile records without intentionally wiping the current database.

Import using Replace

Replace overwrites the relevant current LifeOS data with the imported package. Use only when the backup is known to be complete and correct.

Undo import

LifeOS can retain a pre-import snapshot for recovery. Use Undo promptly if an import produces the wrong result.

Export firstCreate a fresh backup before any Merge or Replace operation.
Inspect the selected fileConfirm it is the intended LifeOS JSON backup, not an older or unrelated file.
Choose Merge or Replace deliberatelyDo not click through the choice without understanding the difference.
Verify counts and critical recordsOpen Diagnostics and manually inspect representative tasks, tags, logs and settings.

19. Diagnostics and data checks

Diagnostics helps confirm what is stored and whether critical datasets are present.

  • Review record counts before and after an import.
  • Check that tasks, logs, tags, settings and English Lab records have plausible counts.
  • Investigate sudden zero counts or unexpected duplication.
  • Do not dismiss storage warnings without understanding the cause.

20. Installation and updates

Install on Android

  1. Open the GitHub Pages URL in Chrome.
  2. Use the browser menu and choose Install app or Add to Home screen.
  3. Launch LifeOS from the new home-screen icon.

Receiving a new version

After a GitHub deployment, reopen or refresh LifeOS. The service worker downloads the new app files and removes older LifeOS caches when the new worker activates.

If an old version remains

  1. Export a backup first.
  2. Close all LifeOS tabs and the installed app.
  3. Reopen and refresh.
  4. Only as a last resort, clear site data—this can delete local LifeOS records.

21. Offline use and background limits

Offline

After a successful online load, the app shell and this guide are cached. Most local features can continue without internet, but external fonts and any future server-based features may not.

Background

Android may suspend LifeOS after you leave it or turn off the screen. A service worker is event-driven, not a permanently running process. Exact alarm delivery and continuous background logic are not guaranteed in the current PWA architecture.

Separate devices

Phone and computer data do not synchronize automatically. Export/import is currently the controlled way to move data, but careless merging can produce duplicates.

How to report a reproducible bug

  1. State the exact page and device.
  2. List the steps from a fresh app opening.
  3. Describe expected versus actual behavior.
  4. Include whether the problem survives restart.
  5. Attach a backup only if it contains no data you are unwilling to share.

23. Practical workflows

Morning execution

Open Today, confirm the day category, check the Next Task, complete medicine/hygiene/breakfast records, and avoid browsing the whole system before beginning.

Workday control

Filter to Career + Pending, start the highest-value task timer, record progress honestly, then clear filters to return to the full day.

Health review

Log meals and measurements, check total sugar/salt rather than only added sugar, and export useful records for a clinician instead of self-adjusting medication.

Evening review

Mark genuine completions and skips, add occurrence notes where the reason matters, review tag balance, and change tomorrow’s system only when the evidence supports it.

Weekly review

Export a backup, inspect repeated skips, remove low-value tasks, review English Lab output and adjust the default week or templates.

24. Operating principles

  • Truth over appearance: accurate skips are more valuable than fake completion.
  • System over motivation: fix recurring friction instead of blaming willpower.
  • Minimum useful complexity: configure only what improves action or insight.
  • Backup before risk: export before imports, resets and major updates.
  • Health humility: record and organize; do not diagnose.
  • Review before expansion: prove daily value before adding more modules.
The correct success metric: LifeOS is useful when it improves real decisions and execution. More tasks, more logs and more modules do not automatically mean a better system.
